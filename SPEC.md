# ReplyHub — 통합 답장 어시스턴트

**OpenAI Build Week Challenge 제출 스펙**
트랙: 당신의 삶을 위한 앱 (Apps for Your Life)
제출 마감: 2026-07-21 (화) 17:00 PT

---

## 1. Problem Statement

사용자는 카카오톡, 위챗, 라인, 텔레그램, 문자메시지 등 5개 안팎의 메신저를 동시에 쓰고 있다. 메시지가 올 때마다 해당 앱을 따로 열어 확인해야 하고, 위챗처럼 외국어(중국어 등)로 오는 대화는 매번 수동으로 번역까지 해야 한다. 답장할 때 상대가 물어본 사실을 모르면 별도로 검색해야 하고, 예전에 나눈 대화 내용을 다시 찾아야 할 때도 앱을 오가며 스크롤해야 한다. 이 앱 전환·번역·검색의 반복이 메시징을 피로한 작업으로 만든다.

해결하지 않을 경우 비용: 답장 지연(관계·업무 영향), 번역 실수로 인한 오해, 과거 대화를 다시 찾는 데 드는 시간 낭비.

## 2. Goals

1. 사용자가 앱을 하나하나 열지 않고 한 피드에서 모든 메신저의 새 메시지를 확인할 수 있다
2. 외국어 메시지는 자동 번역되어 표시되고, 답장도 자동으로 상대방 언어로 변환되어 나간다
3. 답장 작성 시 필요한 사실(웹 검색) 또는 과거 대화 맥락(대화 히스토리 검색)을 AI가 찾아 초안에 반영한다
4. 음성 명령만으로 답장을 부르고 보낼 수 있다 (핸즈프리)
5. 데모 시나리오 3개가 3분 영상 안에서 안정적으로 재현 가능하다 (심사 통과의 최소 기준)

## 3. Non-Goals (v1 / 해커톤 스코프)

- **iOS 지원 안 함** — Apple이 알림 리스너 API를 제공하지 않아 핵심 기능(자동 캡처)이 성립하지 않음. Android 전용으로 한정.
- **실제 크롬/브라우저 앱을 외부 조작하지 않음** — AccessibilityService로 크롬을 흉내내는 대신, 웹 검색은 검색 API 호출 + GPT 요약으로 처리. 훨씬 안전하고 5일 안에 구현 가능.
- **메신저 외 다른 종류의 액션(캘린더, 주문, 송금 등) 자동화하지 않음** — 범용 "폰 조작 에이전트"는 별개의 훨씬 큰 프로젝트. README에 향후 확장 방향으로만 언급.
- **그룹채팅 전용 처리 로직 없음** — 1:1 대화만 우선 지원. 그룹채팅은 알림이 오면 피드에는 뜨지만 답장 어시스턴트 기능은 v1에서 제외.
- **오프라인/저사양 네트워크 대응 없음** — 데모 환경(Wi-Fi 안정) 기준으로만 검증.

## 4. Target User & User Stories

**주 사용자**: 여러 메신저를 동시에 쓰며, 그중 최소 하나는 외국어 대화 상대가 있는 사람 (해외 친구/거래처가 있는 20~40대)

- 사용자로서, 카톡·위챗·라인·텔레그램·문자 알림을 하나의 피드에서 확인하고 싶다. 앱을 매번 갈아타지 않기 위해서.
- 사용자로서, 외국어로 온 메시지를 번역된 상태로 바로 읽고 싶다. 별도 번역 앱을 쓰지 않기 위해서.
- 사용자로서, 답장에 필요한 사실을 모를 때 AI가 검색해서 답장 초안을 만들어주길 원한다. 직접 검색창을 열지 않기 위해서.
- 사용자로서, 예전에 나눈 대화 내용을 다시 찾고 싶을 때 AI가 대화 히스토리에서 찾아주길 원한다. 스크롤하며 직접 찾지 않기 위해서.
- 사용자로서, 손을 쓰지 않고 말로 답장을 부르고 보내고 싶다. 이동 중이거나 손이 바쁠 때도 답장하기 위해서.
- 사용자로서, 특정 앱이 자동 전송을 지원하지 않아도 최소한 번역된 답장을 붙여넣기만 하면 되길 원한다. 기능이 아예 안 되는 것보다 한 단계 더 거치는 게 낫기 때문에.

## 5. Technical Architecture Overview

```
[메신저 알림] → NotificationListenerService (OS 레벨, 앱별 통합 불필요)
                    ↓
            메시지 저장 + 인덱싱 (로컬 DB, 검색용)
                    ↓
        GPT-5.6: 언어 감지 → 번역 → 우선순위 분류
                    ↓
            통합 피드 UI에 표시
                    ↓
    [답장 필요] → 사용자 입력(텍스트 or 음성)
                    ↓
        GPT-5.6 답장 초안 생성
          ├─ 사실 확인 필요 → 웹 검색 API 호출 → 결과 반영
          └─ 과거 맥락 필요 → 로컬 메시지 히스토리 검색 → 결과 반영
                    ↓
            번역 + 톤 조정 (상대방 언어로)
                    ↓
        발송: RemoteInput 지원 앱 → 즉시 전송
             미지원 앱 → 클립보드 복사 + 딥링크로 앱 오픈
```

**핵심 컴포넌트**
- Android `NotificationListenerService`: 알림 텍스트 캡처 (앱 무관, OS 레벨)
- 로컬 저장소: 캡처된 메시지 저장 + 검색 인덱스 (SQLite 등)
- GPT-5.6 API: 번역, 우선순위 분류, 답장 초안 생성, 검색 결과 요약
- OpenAI 내장 `web_search` 도구: 사실 확인용, 브라우저 자동화 아님
- Android `RemoteInput`: 지원 앱 대상 알림 기반 퀵리플라이 전송
- Android `SpeechRecognizer`: 음성 → 텍스트 입력
- 클립보드 + 딥링크(Intent): RemoteInput 미지원 앱 폴백

## 6. Requirements

### Must-Have (P0)
- [ ] NotificationListenerService로 카카오톡, 위챗 최소 2개 앱의 알림 텍스트 캡처
- [ ] 캡처된 메시지를 언어 감지 후 자동 번역하여 통합 피드에 표시
- [ ] 답장 입력 시 GPT가 웹 검색 또는 대화 히스토리 검색 중 필요한 쪽을 판단해 초안 생성
- [ ] 생성된 답장을 상대방 언어로 번역
- [ ] 최소 1개 앱에서 RemoteInput 자동 전송, 나머지는 클립보드+딥링크 폴백 동작
- [ ] 음성 입력으로 답장 텍스트 생성 가능
- [ ] 데모 시나리오 3개가 실제 기기에서 재현 가능

### Nice-to-Have (P1)
- [ ] 라인, 텔레그램, 문자메시지까지 캡처 확장
- [ ] 우선순위 분류(지금 답장 vs 나중에) UI 반영
- [ ] 메시지 히스토리 전문 검색 UI (피드 내 검색창)
- [ ] 답장 톤 커스터마이징 (존댓말/반말 등)

### Future Considerations (P2)
- 범용 폰 액션 에이전트로 확장 (캘린더, 주문 등) — README에 비전으로만 서술
- 그룹채팅 지원
- iOS 대응 (알림 리스너 제약으로 근본적 재설계 필요)

## 7. Acceptance Criteria (핵심 시나리오)

**시나리오 1 — 웹 검색 기반 답장**
- Given 위챗에서 중국어로 사실 확인 질문(예: 영업시간)이 옴
- When 사용자가 피드에서 해당 메시지에 답장을 요청함
- Then GPT가 필요성을 판단해 웹 검색을 수행하고, 검색 결과를 반영한 답장 초안을 중국어로 생성함

**시나리오 2 — 대화 히스토리 기반 답장**
- Given 카카오톡 친구가 과거에 공유했던 정보(예: 주소)를 다시 물어봄
- When 사용자가 답장을 요청함
- Then GPT가 로컬 메시지 히스토리에서 관련 내용을 검색해 답장 초안에 반영함

**시나리오 3 — 음성 답장**
- Given 사용자가 마이크 버튼을 누르고 한국어로 답장 내용을 말함
- When 음성 인식이 완료됨
- Then 텍스트로 변환 → 상대방 언어로 번역 → RemoteInput 또는 클립보드+딥링크로 발송됨

## 8. Success Metrics (해커톤 맥락)

이 프로젝트는 사용자 지표보다 **심사 기준 통과 여부**가 핵심 성공 지표다.

- 기술적 구현: P0 요구사항 전체가 실제 기기에서 작동 (에뮬레이터 아님 권장)
- 디자인: 3개 시나리오가 매끄러운 하나의 제품 경험으로 이어짐 (기능 나열이 아니라)
- 잠재적 영향: README/데모에서 "여러 메신저 + 다국어 소통"이라는 구체적이고 흔한 문제를 명확히 제시
- 아이디어 참신성: "완전 자동 통합 클라이언트"가 아니라 "우아하게 성능이 저하되는 트리아지 허브"라는 현실적 접근을 명시적으로 강조

## 9. Risks & Open Questions

**가장 리스키한 가정 (Day 1 스파이크로 즉시 검증 필요)**
- 카카오톡, 위챗, 라인 알림이 실제로 RemoteInput 퀵리플라이 액션을 노출하는가? → 미노출 시 해당 앱은 클립보드+딥링크로만 동작 (스코프 내 이미 대응됨)
- NotificationListenerService가 각 앱의 전체 메시지 텍스트를 안정적으로 캡처하는가? (일부 앱은 민감 알림으로 내용을 축약해서 보낼 수 있음) → 엔지니어링 확인 필요, 미확보 시 해당 앱은 v1 지원 목록에서 제외

**오픈 퀘스천**
- OpenAI 내장 `web_search`의 데모 환경 지연과 호출 비용이 허용 범위인지 확인 필요 (non-blocking)
- 로컬 메시지 저장이 개인정보 관점에서 문제없는지 (기기 로컬에만 저장, 외부 전송 안 함으로 스펙에 명시할 것 — 데모/README에서 프라이버시 스토리로 언급)
- Day 1 스파이크 결과에 따라 카카오톡/위챗 외 추가 앱(P1)을 시도할지 여부는 스파이크 이후 결정 (non-blocking)
